## 一、评测在测什么

大模型的能力评测大致分成几个维度，每个维度对应不同的题型和评分方式。

| 维度    | 代表 Benchmark              | 题型             |
| ----- | ------------------------- | -------------- |
| 知识与推理 | MMLU、C-Eval、BBH           | 选择题、多学科考题      |
| 数学    | GSM8K、MATH                | 应用题、竞赛题        |
| 代码    | HumanEval、SWE-bench       | 函数实现、真实仓库修 bug |
| 长文本   | LongBench                 | 长上下文问答、摘要      |
| Agent | AgentBench、Terminal-Bench | 多步操作任务         |
| 安全对齐  | M3-SafetyBench、SaladBench | 有害内容识别、越狱测试    |
| 主观偏好  | Chatbot Arena             | 人类两两投票         |

这几类分数之间的相关性并不高。一个数学能力强的模型，安全对齐未必做得好；一个代码能力强的模型，长文本理解能力也不一定强。所以判断一个模型强不强，要先选定维度。

## 二、评测数据长什么样

一条评测数据通常包含三部分：输入（题目）、参考答案（或评分标准）、元信息（学科、难度等，方便后续按维度拆分统计）。以 MMLU 风格的选择题为例，一条原始数据大概是这样：

```
{
  "id": "mmlu_astronomy_0187",
  "subject": "astronomy",
  "question": "为什么天空是蓝色的？",
  "choices": [
    "A. 大气分子对短波长光的散射更强",
    "B. 大气分子吸收了蓝光并重新发射",
    "C. 海洋反射蓝光到天空中",
    "D. 太阳本身发出的光就是蓝色"
  ],
  "answer": "A",
  "difficulty": "medium"
}
```

代码类 Benchmark 的数据结构不太一样，参考答案往往不是一个固定字符串，而是一段可执行的测试代码：

```
{
  "task_id": "HumanEval/42",
  "prompt": "def add_one(nums: List[int]) -> List[int]:\n    """给列表中每个元素加一"""\n",
  "test": "assert add_one([1, 2, 3]) == [2, 3, 4]\nassert add_one([]) == []",
  "entry_point": "add_one"
}
```

Agent 类 Benchmark 的数据更接近一条完整的任务脚本，包含环境初始化、允许调用的工具、终止条件，篇幅比前两种长得多，这里不展开。

## 三、评测集的构建

构建思路大致有四种，成本和质量各不相同：

**人工出题**：领域专家按能力大纲手写，质量最高，成本也最高，MMLU、C-Eval 早期都是这么来的。

**从真实数据里筛选**：比如 SWE-bench 直接从 GitHub 上已合并的真实 PR 里挑任务，场景真实，也不容易被"背答案"。

**LLM** **生成 + 人工审核**：先用大模型批量生成候选题目，再由人工筛选修正，效率比纯人工高不少，是目前构建新 Benchmark 的主流方式之一。一个简化的生成流程大概是这样：

**合成数据**：适合冷启动、增加多样性、覆盖边缘情况、对抗性测试，以及 RAG 场景评估。比如做 RAG 评测时，可以用知识库文档反推问题：

```
def synthesize_rag_qa(document_chunk: str, model_client):
    prompt = f"""基于下面这段文档内容，生成一个只能用这段内容回答的问题，以及对应的标准答案。
文档：{document_chunk}
输出格式：{{"question": "...", "answer": "...", "source_chunk": "文档片段原文"}}"""
    return parse_json(model_client.generate(prompt))
```

构建时有两个坑需要特别注意。一是难度分布，题目不能全是送分题，也不能全是刁钻题,要有梯度，否则区分不出模型之间的差距。二是数据污染：题目是否可能已经出现在模型的训练语料里。如果出现过，模型答对靠的是记忆而不是推理，分数就失真了。像 LiveCodeBench 这类 Benchmark 应对的办法是持续收录训练截止日期之后新发布的题目，用时间窗口来防止污染。

## 四、打分方法

1.  ### 规则打分（Rule-based）

适合有唯一正确答案的任务，客观、便宜、可复现。选择题直接比对字母，代码题跑单元测试，数学题解析最终数值。

代码能力最常用的指标是 **pass@k**：给模型 $$$$ 次采样机会，只要其中至少 1 次通过测试就算这道题过关，公式是

$$\text{pass@}k = \mathbb{E}_{\text{problems}}\left[1 - \frac{\binom{n-c}{k}}{\binom{n}{k}}\right] $$

其中 $$$$ 是总采样次数，$$$$ 是这 $$$$ 次里通过测试的次数，$$$$ 是允许的尝试次数。这个公式看起来复杂，实际上是为了避免直接用 $$$$ 次采样估算时方差过大，用组合数做了无偏估计。（更详细的解释见：[大模型评测指标详解](https://nenocat.github.io/%E5%A4%A7%E6%A8%A1%E5%9E%8B%E8%AF%84%E6%B5%8B%E6%8C%87%E6%A0%87%E8%AF%A6%E8%A7%A3/)）

2.  ### 人工打分（Human Evaluation）

找标注员按评分标准打分，或做两两对比（回答 A 和回答 B 哪个更好）。最贴近真实的人类判断，但成本高、速度慢，标注者之间还存在一致性问题。衡量标注者一致性常用 Cohen's Kappa：

$$\kappa = \frac{p_o - p_e}{1 - p_e} $$

其中 $$p_$$ 是标注者之间实际观察到的一致比例，$$p_$$ 是随机情况下预期的一致比例，例如任务是"回答 A 好还是 B 好"这种二选一,且两人选 A 的概率都是 50%,那纯靠猜,两人猜出同一个答案的概率就有 50%，那这时 $$p_$$ 为0.5。$$\kapp$$ 越接近 1，说明标注标准越清晰、标注者理解越一致；如果 $$\kapp$$ 偏低，往往说明评分标准本身写得不够明确，需要回头修改标注指南，而不是急着换标注员。

3.  ### 模型打分（LLM-as-a-Judge）

用另一个能力较强的模型来给答案打分或做两两对比，替代人工，是目前处理开放式生成任务（比如写作、对话、摘要）最常用的方式，因为规则打分处理不了"这段话写得好不好"这种问题，人工打分又跑不动大规模的评测。

一个简化的 pointwise 打分 prompt 大概长这样：

```
你是一名严格的评审员，请根据下面的评分标准给这个回答打 1-5 分。

【用户问题】
{question}

【模型回答】
{answer}

【评分标准】
5分：完全回答问题，事实准确，逻辑清晰
3分：基本回答问题，但存在小的遗漏或表述不清
1分：答非所问，或包含事实错误

请只输出一个 JSON：{"score": <1-5的整数>, "reason": "简要理由"}
```

pairwise（两两对比）的写法更常见，因为绝对打分容易受评委自身标准漂移的影响，相对比较更稳定：

```
def llm_judge_pairwise(question, answer_a, answer_b, judge_model):
    prompt = f"""下面有同一个问题的两个回答，请判断哪个更好。
问题：{question}
回答A：{answer_a}
回答B：{answer_b}
只输出 "A" 或 "B" 或 "平局"，不要解释。"""
    # 建议正反各跑一次交换 A/B 顺序，抵消模型的位置偏好
    result_1 = judge_model.generate(prompt)
    prompt_swapped = prompt.replace(answer_a, "TEMP").replace(answer_b, answer_a).replace("TEMP", answer_b)
    result_2 = judge_model.generate(prompt_swapped)
    return result_1, result_2
```

代码里"正反各跑一次交换顺序"这一步不是多余的操作，而是因为已有研究发现，无论是人类评委还是 LLM 评委,都容易受到不同类型偏见的影响,其中一种就是位置偏见——即便两个回答质量相同，排在前面的选项更容易被判定为更好。除了位置偏见，还有长度偏见（偏爱更长的回答）和自我偏好偏见（模型评委更偏爱风格和自己相似的回答）。即便是当前最强的模型评委，这些偏见也没有被完全消除。

比较严谨的做法是额外构建一个小规模的人工标注校准集（通常一百到五百条），用统计方法修正 LLM 评委的系统性偏差，并在报告结果时附上置信区间，而不是只报告一个原始准确率。

**Rubric** **与 Criterion**

前面的 pointwise 打分 prompt 里，"评分标准"只有一条笼统的描述（回答得好不好），这在题目简单时够用，但遇到复杂的开放式任务——比如写一份完整的病情说明、评估一次多轮客服对话——单一分数很难说清楚回答到底哪里有问题。这时候通常会引入 **criterion（评分维度）** 和 **rubric** **（评分细则）** 这两个概念。

一个 criterion 是评分时关注的一个具体方面，比如"事实准确性""内容完整性""语气是否得体""是否包含安全隐患"。一个 rubric 就是把多个 criterion 组织在一起、并且给每个 criterion 标注好"打几分对应什么表现"的完整评分表。写 rubric 时有几条常见的经验：

-   **每一档分数都要有具体描述**，不能只写"1-5分"，而要写清楚 3 分对应什么样的回答、1 分对应什么样的回答，否则不同评委（不管是人还是模型）对同一个"3分"的理解会不一样；
-   **每一条 criterion 尽量用行为化的语言描述**，比如写"包含虚假信息扣分"，而不是写"质量低扣分"，后者太抽象，评委没有可操作的判断依据；
-   **criterion 数量不宜过多**，三到五条是比较常见的区间，条目太多会分散评委的注意力，也不会带来更多有效信号；
-   **不同 criterion 的重要性通常不一样**，需要加权而不是简单平均，比如一个医疗问答场景里，"事实准确性"出错的代价远高于"语气生硬"，权重就应该差出好几倍。

一条包含多个 criterion 的 rubric 数据大概长这样：

```
{
  "task_id": "support_bot_042",
  "criteria": [
    {
      "name": "accuracy",
      "weight": 0.5,
      "levels": {
        "5": "回答的所有事实性信息都准确，且与问题直接相关",
        "3": "主体信息准确，但存在次要细节错误或遗漏",
        "1": "包含会误导用户的事实性错误"
      }
    },
    {
      "name": "completeness",
      "weight": 0.3,
      "levels": {
        "5": "完整覆盖用户问题涉及的所有子问题",
        "3": "覆盖主要问题，遗漏个别次要子问题",
        "1": "只回答了部分问题，或答非所问"
      }
    },
    {
      "name": "tone",
      "weight": 0.2,
      "levels": {
        "5": "语气得体，符合客服场景规范",
        "3": "语气基本得体，但略显生硬",
        "1": "语气不当，可能引起用户不满"
      }
    }
  ]
}
```

打分时让评委模型对每个 criterion 单独给分，再按权重汇总成一个总分，而不是让模型直接给一个笼统的总分：

```
def rubric_based_score(question, answer, rubric, judge_model):
    per_criterion_scores = {}
    for criterion in rubric["criteria"]:
        prompt = f"""请只依据下面这一条评分维度给回答打分，不要考虑其他方面。

【维度】{criterion['name']}
【评分档位】
{format_levels(criterion['levels'])}

【用户问题】{question}
【模型回答】{answer}

只输出 JSON：{{"score": <对应档位的数字>, "evidence": "支持该打分的具体依据"}}"""
        result = parse_json(judge_model.generate(prompt))
        per_criterion_scores[criterion["name"]] = result["score"]

    weighted_total = sum(
        per_criterion_scores[c["name"]] * c["weight"] for c in rubric["criteria"]
    )
    return weighted_total, per_criterion_scores
```

这种"拆开打分、加权汇总"的做法比让模型直接打一个总分更稳定，原因和前面提到的 pairwise 优于 pointwise 是同一个道理：给模型的判断空间越窄、越具体，输出的一致性就越高。

拆分打分还有一个额外的好处，就是能看到 per-criterion 的分数明细——如果下一版模型总分下降了，明细能直接告诉你是哪个维度退步了，而不需要再去猜。OpenAI 的 HealthBench 是这类方法里规模比较大的一个例子，它由医生撰写了数万条针对具体对话场景的评分标准，让评委模型逐条检查回答是否满足每条标准，再把命中的加权分数汇总成最终得分。

需要注意的是，rubric 打分同样会受到前面提到的长度偏见和自我偏好偏见的影响,只是因为拆分成了多个维度、每个维度的判断空间更窄，偏见造成的偏差通常比笼统打一个总分要小一些，而不是完全消失。

5.  ### 主观偏好排名：ELO 分数

Chatbot Arena 这类"人类投票"排行榜用的是国际象棋里的 ELO 评分系统。两个模型对战，胜者分数上升,败者分数下降,涨跌幅度取决于赛前的分差,分差越大,爆冷的一方分数变化越剧烈:

$$E_A = \frac{1}{1 + 10^{(R_B - R_A)/400}} $$

$$R_A' = R_A + K \cdot (S_A - E_A) $$

其中 $$R_A$$、$$R_B$$ 是对战双方赛前分数，$$E_A$$ 是模型 A 的预期胜率，$$S_A$$ 是实际比赛结果（赢记 1，输记 0，平局记 0.5），$$K$$ 是调节分数变化幅度的常数。这套机制的直觉是：强模型战胜弱模型只能拿到很少的分数，弱模型爆冷战胜强模型则能拿到大量分数，符合人类直觉里的合理性。（更详细的解释见：[大模型评测指标详解](https://nenocat.github.io/%E5%A4%A7%E6%A8%A1%E5%9E%8B%E8%AF%84%E6%B5%8B%E6%8C%87%E6%A0%87%E8%AF%A6%E8%A7%A3/)）
